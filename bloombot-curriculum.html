Unit 01

Orienting to the Codebase

Before reading a single line of code, learn how professional developers approach an unfamiliar project. You'll map the repository's file system, identify naming conventions, and build a mental model of how the pieces connect.

📖 Concepts What you'll learn ▶
Software Architecture Project Management Coding

Key ideas

Every codebase tells a story through its file structure. Bloombot is a static site — there is no database and no server-side rendering. Every page is a self-contained HTML file that the browser opens directly. This is the simplest possible web architecture, and understanding why it works (and where it breaks down) is your first architectural lesson.

You'll also encounter the concept of naming conventions. Notice that Bloombot uses two patterns: hyphenated names (bloombot-modern.html) for "structural" pages, and descriptive names with spaces (bloombot - Lightning.html) for "content" pages. This is a design choice — and arguably a mistake — that has real consequences for URLs and command-line usage.

Files to explore

  • .gitignore — What's being hidden and why? Note the comment about "embedded API keys" in the original Google Sites export.
  • CNAME — A single line: bloombot.com. This tells GitHub Pages which custom domain to serve.
  • app.py — The local development server. Just 41 lines of Python, but it introduces inheritance (class Handler extends SimpleHTTPRequestHandler), port binding, and the concept of a dev server vs. production hosting.
  • webpage-main/ — The artifact library. Run ls webpage-main/ and count the files. Each .html file is a standalone interactive tool. Each .stl file is a 3D model.

Architecture concept: Static vs. Dynamic sites

Bloombot has no backend logic. When a user visits bloombot.com/bloombot-i101.html, GitHub's servers simply hand them the file. Contrast this with a dynamic site like Twitter, where every page load triggers server-side code that queries a database and assembles HTML on the fly. Static sites trade interactivity limitations for simplicity, speed, and free hosting.

Unit 02

HTML & the Document

Read the HTML of Bloombot's pages closely. Understand how a document is structured, what semantic tags communicate, and how the browser turns markup into a rendered page.

📖 Concepts Document structure & semantics ▶
Coding

The anatomy of a Bloombot page

Open bloombot-modern.html in a text editor. The first line — <!DOCTYPE html> — is a declaration that tells the browser to use modern HTML5 parsing rules. Everything else lives inside the <html> root element, split into two children: <head> (metadata the user never sees) and <body> (everything visible).

Inside the <head>, study these elements:

  • <meta charset="UTF-8"> — character encoding. Without this, special characters can break.
  • <meta name="viewport"> — responsive design instruction for mobile browsers.
  • <link> tags — loading Google Fonts externally. Notice how rel="preconnect" tells the browser to start the network handshake early, before the font file is actually needed.
  • <style> — all CSS is inline rather than in a separate .css file. This is an architectural choice worth questioning.

Semantic HTML vs. div soup

Bloombot uses <header>, <nav>, <section>, and <footer> — semantic tags that carry meaning for screen readers and search engines. Compare this to a page built entirely from <div> tags: the structure would look identical visually but would be meaningless to assistive technology.

File to study

bloombot-i101.html is the richest page to study. It has a hero section, a metadata band, eight topic sections, artifact card grids, a slides callout band, and a footer — all within a single HTML document.

Unit 03

CSS & Visual Design Systems

Bloombot has a sophisticated visual identity built entirely with CSS custom properties. Learn how design tokens, the box model, and CSS Grid create a consistent, maintainable look.

📖 Concepts Design tokens & layout systems ▶
Coding Architecture

CSS custom properties as design tokens

At the top of every Bloombot page, you'll find a :root block defining the site's visual vocabulary:

:root { /* Neutral palette */ --ink: #1a1612; --cream: #f5f0e8; --warm: #e8dcc8; --rust: #c4541a; --amber: #d97706; /* Per-course accent colors */ --c101: #1d4ed8; /* blue — informatics */ --c341: #b45309; /* amber — electronics */ --c549: #6d28d9; /* violet — prototyping */ }

In professional design systems, these are called design tokens — named values that encode visual decisions in one place so they can be changed globally. If you wanted to rebrand the site from warm cream to cool gray, you'd change about six lines.

CSS Grid and responsive design

The artifact card grid in bloombot-i101.html uses a single powerful line:

grid-template-columns: repeat(auto-fill, minmax(220px, 1fr));

This says: "Fill the row with as many columns as will fit, where each column is at least 220px wide and stretches equally to fill remaining space." This single declaration handles responsiveness — from 4 columns on desktop to 1 on mobile — without a single media query.

The architectural problem: duplicated styles

Every HTML file in Bloombot contains its own <style> block, and many of them repeat the same CSS. This is a direct consequence of the "every page is self-contained" architecture. Compare this to a shared styles.css file that every page links to. What are the trade-offs of each approach?

🛠️ Exercise Extract and refactor the design system ▶

  4. Study the responsive breakpoint at the bottom of bloombot-i101.html: @media (max-width:680px). What three things change? Resize your browser window to see them in action.
Unit 04

JavaScript Fundamentals

Move from reading static markup to understanding how JavaScript brings pages to life. Study real event listeners, DOM manipulation, and state management in Bloombot's interactive artifacts.

📖 Concepts DOM manipulation & event-driven programming ▶
Coding

The simplest JavaScript in Bloombot

Look at the security-audit-2026-03-01.html file. It has a collapsible findings UI built with minimal JavaScript:

document.querySelectorAll('.finding-header').forEach(function(header) { header.addEventListener('click', function() { this.parentElement.classList.toggle('open'); }); });

This pattern — select elements, attach listeners, toggle classes — is the bread and butter of vanilla JavaScript. The CSS does the visual work (showing/hiding via .open .finding-body { display: block; }), and the JavaScript just flips a switch. This separation of concerns is a core architectural principle.

A richer example: the password analyzer

Open webpage-main/password_analyzer.html. This artifact takes user input (a password), calculates entropy, checks for common patterns, and renders a strength meter — all in real time, all client-side. Study how:

  • An input event fires on every keystroke (not just when the user presses Enter).
  • Multiple functions each assess one aspect of password strength.
  • The results are composed together and the DOM is updated.

Understanding scope and closures

In nav.js, the entire script is wrapped in (function() { ... })(); — an Immediately Invoked Function Expression (IIFE). This creates a private scope so that variables like menus and prefix don't leak into the global namespace and conflict with other scripts. This is a foundational JavaScript pattern worth understanding deeply.

Unit 05

Architecture & Code Reuse

Zoom out from individual pages and study how the codebase handles (and fails to handle) shared concerns. The navigation system is a case study in practical code architecture.

📖 Concepts The nav.js pattern & DRY principle ▶
Architecture Coding Project Management

Single source of truth

nav.js is the most architecturally important file in the repo. It solves a real problem: if you have 60+ HTML files and you want to add a navigation link, you shouldn't have to edit every single file. Instead, each page just declares:

<nav id="main-nav" data-active="i101"></nav> <script src="nav.js"></script>

The data-active attribute tells nav.js which menu item to highlight. The script auto-detects its own depth (is it loaded as nav.js or ../nav.js?) and prefixes all links accordingly. This is a convention-over-configuration pattern — the system infers context from file location rather than requiring explicit setup.

What nav.js doesn't solve

Notice that nav.js only handles the navigation bar. The CSS for the nav is still duplicated in every file's <style> block. The footer is also repeated everywhere. This is a textbook case of technical debt — it works, but future changes require editing many files. In a framework like React, you'd have a <Layout> component that wraps every page. Without a framework, what are your options?

Architectural patterns to recognize

  • Data-driven rendering: nav.js defines a menus array (data) and a rendering function (logic). Changing the menu means changing data, not rewriting HTML. This is the same principle that powers React, D3, and every data-driven UI.
  • IIFE encapsulation: The entire file is wrapped in (function() { ... })();, creating a private scope. No globals leak out.
  • Graceful degradation: The if (!nav) return; guard at line 10 means the script silently does nothing if a page forgets to include the nav element, rather than crashing.

Task

  2. Add a new top-level menu item to the menus array in nav.js — perhaps "Curriculum" pointing to your new curriculum page. Reload three different pages and verify the new link appears everywhere.
  3. Build something: Create a footer.js file following the same pattern as nav.js. It should inject a consistent footer into any page that includes <footer id="main-footer"></footer> and the script. Test it on two pages.
  4. Architectural analysis: Write a half-page comparing the nav.js "inject from JavaScript" approach against three alternatives: (a) a shared CSS file with <link>, (b) a static site generator like Jekyll or Hugo, (c) a component framework like React. What would you gain and lose with each?
Unit 06

Data Visualization with D3.js

Bloombot uses D3.js for its most complex artifacts — force-directed network graphs, topology visualizations, and interactive data displays. Learn to read and modify data-driven graphics.

📖 Concepts Data binding, selections & force simulations ▶
Coding Architecture

D3's mental model

D3 (Data-Driven Documents) doesn't draw charts for you — it binds data to DOM elements and lets you describe transformations. Open spatial-network-viz.html (953 lines) and find the force simulation setup. D3's force layout follows these steps:

  1. Define nodes (entities) and links (relationships) as data arrays.
  2. Create a simulation with forces: charge (repulsion), centering, link distance, and collision avoidance.
  3. On each simulation tick, update SVG element positions to match the calculated coordinates.
  4. Add drag behavior so users can interact with nodes.

SVG as the rendering target

Unlike the HTML-based pages, the network visualizations render into <svg> — Scalable Vector Graphics. SVG elements (<circle>, <line>, <text>) are part of the DOM and can be styled with CSS and manipulated with JavaScript, but they use a coordinate system rather than the document flow model.

Files to study

  • spatial-network-viz.html — The most complex D3 artifact. Includes a legend, detail panel, category-based coloring, and pinning behavior.
  • webpage-main/topology.html — Network topology visualization.
  • webpage-main/network_analysis.html — Analytical view of network structures.
  • webpage-main/network_strategic_link.html — Strategic link analysis.
🛠️ Exercise Read and modify a D3 visualization ▶

Task

  1. Open spatial-network-viz.html in your browser. Interact with it — drag nodes, click on them, observe the detail panel. Then open the source code and identify: Where is the data defined? Where is the simulation created? Where does the tick function update positions?
  2. Find the color scale that assigns colors to different node categories. Change one of the colors and reload. This gives you a feel for how data maps to visual encoding.
  3. Study the drag behavior. How does D3 handle the start, during, and end of a drag gesture? Why does it "reheat" the simulation on drag start?
  4. Build something: Create a minimal force-directed graph from scratch: 5 nodes connected by 5 edges. Use the D3 CDN (https://d3js.org/d3.v7.min.js). Apply Bloombot's color tokens to your nodes. It doesn't need to be fancy — the goal is to understand the simulation lifecycle.
Unit 07

3D Graphics & ES Modules

Bloombot includes a 3D model viewer built with Three.js. This unit introduces WebGL rendering, ES module imports, and the scene/camera/renderer pattern used across all 3D web applications.

📖 Concepts Three.js architecture & ES modules ▶
Coding Architecture

The scene graph pattern

Open webpage-main/stlM2.html. Every Three.js application follows the same architecture:

  1. Scene — the container for all 3D objects, lights, and cameras. Think of it as the stage.
  2. Camera — defines the viewpoint (perspective vs. orthographic, field of view, near/far clipping planes).
  3. Renderer — takes the scene and camera, and draws pixels to a <canvas> element using WebGL.
  4. Animation loop — a recursive requestAnimationFrame call that re-renders the scene every frame (~60 times per second).

ES Modules vs. script tags

The 3D viewer uses import statements:

import * as THREE from './three.module.js'; import { STLLoader } from './STLLoader.js'; import { OrbitControls } from './OrbitControls.js';

This is the ES Module system — JavaScript's native way to share code between files. It requires a <script type="module"> tag, and crucially, it only works when served over HTTP (not when double-clicking the file). This is exactly why app.py exists — to provide a local HTTP server for development.

Why does the 3D viewer need a server?

Browsers block ES module imports from file:// URLs for security reasons (CORS policy). The STL loader also uses fetch() to load 3D model files, which has the same restriction. This is a real-world constraint you'll encounter in many web projects.

  3. Find the lighting setup. What types of lights are used? What happens if you remove the ambient light? (Try it.) What happens if you change the directional light's position?

Unit 08

Testing, Auditing & Quality Assurance

Bloombot has no automated test suite — which is itself a lesson. Learn about the testing strategies that should exist, study the security audit that does, and think about what "quality" means for a static educational site.

📖 Concepts Testing levels, security audits & accessibility ▶
Testing Project Management

The testing gap

Bloombot has zero automated tests. No unit tests, no integration tests, no end-to-end tests. For a solo educational project, this is understandable — but it means any change could silently break something. The testing pyramid is a useful mental model:

  • Unit tests — test individual functions in isolation. For Bloombot, this might mean testing that the password entropy calculation in webpage-main/password_analyzer.html returns correct values for known inputs.
  • Integration tests — test that components work together. Does nav.js correctly generate links for pages at different directory depths?
  • End-to-end tests — test the full user experience. Does clicking the "Networks" link on the course page actually navigate to the right section?

The security audit as a testing artifact

Open security-audit-2026-03-01.html. This is a manual audit organized by severity (Critical, High, Medium, Low, Info, Pass). Study its structure — each finding has a title, severity level, description, and recommended fix. This is a real-world security review format. Notice how it flags issues like:

  • Missing Content Security Policy headers
  • Inline scripts (which a strict CSP would block)
  • External resource loading without integrity checks

For a GitHub Pages static site, many server-side security headers aren't configurable — but the audit is still valuable for understanding what would matter on a real server.

Accessibility as a quality dimension

Run a browser accessibility audit (Chrome DevTools → Lighthouse → Accessibility) on bloombot-i101.html. How does it score? Bloombot uses semantic HTML (good), but it may lack ARIA labels, alt text, and sufficient color contrast ratios in some areas.

Unit 09

Dev Tooling & Deployment

Understand the tools and workflows that take code from your editor to a live website. Git version control, the Python dev server, GitHub Pages hosting, and the decisions that shaped this project's toolchain.

📖 Concepts Git, local servers & static hosting ▶
Project Management Architecture

Git as project memory

Run git log --oneline in the Bloombot repo. Each commit is a snapshot of the entire project at a point in time. Notice the commit messages — they tell a story of the project's evolution. Good commit messages are a form of documentation that the .gitignore is a form of security: it prevents the original Google Sites export (which contained embedded API keys) from ever being pushed to GitHub.

The deployment pipeline (or lack thereof)

Bloombot's deployment is as simple as it gets: push to the main branch on GitHub, and GitHub Pages automatically serves the files at bloombot.com. The CNAME file tells GitHub which custom domain to use. There is no build step, no compilation, no bundling. This is the ultimate "what you see is what you deploy" workflow.

Compare this to a modern web project that might require: installing dependencies (npm install), running a build (npm run build), running tests (npm test), and only deploying if all checks pass. That's a CI/CD pipeline (Continuous Integration / Continuous Deployment). Bloombot's simplicity is a feature — but it also means there's no safety net between "git push" and "live on the internet."

The local server: app.py in depth

app.py is 41 lines of Python that teaches several concepts:

  • Class inheritance: Handler extends SimpleHTTPRequestHandler, overriding just one method (log_message) to suppress non-error output. This is the Open/Closed Principle in action — extending behavior without modifying the base class.
  • Port binding: TCPServer(("", PORT), Handler) tells the operating system to listen on port 8000 for HTTP connections.
  • Context manager: The with statement ensures the server is properly cleaned up even if an error occurs.
  • Developer ergonomics: webbrowser.open(url) automatically opens the browser — a small but thoughtful touch.
🛠️ Exercise Master the toolchain ▶

  2. Create a new branch (git checkout -b my-experiment), make a small change to any page, commit it, then switch back to main. Verify your change disappeared. Switch back to my-experiment — it's back. This is the fundamental Git workflow for safe experimentation.

You don't need to implement it — just describe the .github/workflows/deploy.yml file you'd create and what each step would do.
Unit 10

Project: Ship a Feature

Everything comes together. Choose a real improvement to Bloombot, plan it like a software project, build it using the skills from Units 1–9, test it, and prepare it for deployment. This is what professional software development feels like.

📖 Concepts The full software development lifecycle ▶
Project Management Architecture Coding Testing

Planning: scope a feature

Professional developers don't just start typing. They write a brief plan that covers: what the feature does, who it's for, what files need to change, what new files are needed, how to test it, and what could go wrong. This is sometimes called a "technical design document" or an "RFC" (Request for Comments).

Choose your project

Select one of these features to implement (or propose your own of similar scope):

  1. Dark mode toggle — Add a button to the nav that switches between light and dark themes using the existing CSS custom property system. Persist the choice in localStorage. Requires modifying: nav.js, every page's CSS tokens, and potentially creating a new theme.js.
  2. Search page — Create a search page that lets users find artifacts by keyword. Build a JSON index of all artifacts (title, description, category, URL) and filter it on the client side as the user types. This exercises data structures, DOM manipulation, and UX design.
  3. Artifact usage tracker — Add a small analytics system that records which artifacts are opened and displays the data on a dashboard page using D3.js. Store counts in localStorage. This combines D3, JavaScript events, and data management.
  4. Shared CSS extraction — Refactor the duplicated CSS across all main pages into a single bloombot.css file. Update every page to reference it. Verify nothing breaks visually. This is a pure architecture and quality task.

The project management arc

Your project should follow these stages, each producing a deliverable:

  • Planning — A written plan (1 page) describing the feature, affected files, and approach.
  • Implementation — The actual code changes on a Git branch.
  • Testing — A manual test checklist (at minimum) or automated tests that verify the feature works. Run Lighthouse before and after.
  • Code review — Write a self-review: What decisions did you make? What trade-offs? What would you do differently?
  • Documentation — Update or create any necessary documentation (a README section, inline code comments, or a changelog entry).
🛠️ Exercise Plan, build, test, reflect ▶

Task

  1. Plan (Day 1): Write your technical plan. Include: feature description, list of files to create or modify, risks or unknowns, and a definition of "done." Share this plan with a classmate or mentor before starting to build.
  2. Branch & build (Days 2–4): Create a feature branch. Commit early and often with clear messages. Aim for small, reviewable commits rather than one giant commit at the end.
  3. Test (Day 5): Run your manual test checklist. Run Lighthouse on affected pages. Run the HTML validator. If you wrote automated tests, run those too. Fix anything that's broken.
  4. Review & reflect (Day 6): Write a self-review (1 page) answering:
    • What was harder than expected?
    • What did the AI-generated code do well, and where did its patterns create friction?
    • If you were starting this project from scratch, what architectural choices would you make differently?
    • What would you want to learn next?
Unit 11

Human-Computer Interaction

Bloombot is ultimately something people use — which makes it a user interface. Step back from the code and study the design principles, mental models, and heuristics that determine whether a page actually works for its audience.

📖 Concepts Design heuristics, affordances & mental models ▶
HCI Architecture

What is HCI?

Human-Computer Interaction is the study of how people interact with technology and how to design systems that are effective, efficient, and satisfying to use. For Bloombot, this means asking: Can a student actually navigate the site, find what they need, and learn from the content — or does the interface get in the way?

Nielsen's 10 usability heuristics

Jakob Nielsen's heuristics are the most widely used framework for evaluating interface design. Apply each one to Bloombot:

  • Visibility of system status — Does Bloombot tell users where they are? The data-active attribute in nav.js highlights the current section, but is it visually obvious enough?
  • Match between system and real world — Does the vocabulary make sense to an information science student? Or does it assume too much prior coding knowledge?
  • User control and freedom — Can a user easily undo or go back? Is there always a clear way home?
  • Consistency and standards — Do all pages follow the same layout pattern? Check whether the collapsible blocks in bloombot-i101.html behave the same way as those in bloombot-scaffolds.html.
  • Error prevention — For interactive elements (like the gear generator), does the interface prevent invalid inputs, or does it fail silently?
  • Recognition rather than recall — Are navigation options visible, or do users have to remember URLs?
  • Flexibility and efficiency of use — Are there shortcuts for experienced users? Could there be?
  • Aesthetic and minimalist design — Does every visual element serve a purpose, or is there unnecessary decoration?
  • Help users recognize, diagnose, and recover from errors — When something doesn't load or a link is broken, what does the user see?
  • Help and documentation — Is there any guidance for first-time visitors on how to use the site?

Affordances and signifiers

An affordance is what an element allows a user to do; a signifier is the visual cue that communicates that affordance. In Bloombot, the collapsible blocks use a chevron (▶) as a signifier for "click to expand." But look at the block headers — is the entire bar clickable, or just the chevron? Test it. If the affordance is broader than the signifier suggests, that's a design gap.

Mental models

Users arrive with expectations about how websites work. A student coming from Canvas or Blackboard has a mental model of "course → modules → content." Does Bloombot's structure match that model? Open bloombot-i101.html and map its information architecture: how many clicks does it take to get from the homepage to a specific artifact?

Unit 12

User Testing & Evaluation

Heuristic evaluation tells you what experts think — user testing tells you what actually happens. Learn to plan, conduct, and analyze usability studies, then use the findings to drive real improvements to Bloombot.

📖 Concepts Usability testing methods & think-aloud protocol ▶
HCI Testing Project Management

Why test with real users?

You can read every HCI textbook and still be surprised by what users actually do. Usability testing reveals the gap between what you designed and what people experience. Even testing with 5 participants uncovers roughly 80% of usability problems (Nielsen & Landauer, 1993). For a site like Bloombot, where the audience is students encountering technical content for the first time, this gap can be enormous.

Think-aloud protocol

The most common usability testing method is the think-aloud protocol: ask a participant to perform tasks while narrating their thought process. As the facilitator, you observe and take notes — you don't help, explain, or lead. Key rules:

  • Write task scenarios, not instructions. Say "You want to find the artifact about network security" — not "Click on the Networks link in the nav bar."
  • Recruit representative users. For Bloombot, this means information science students who have minimal web development experience — not your CS friends.
  • Record everything. With permission, screen-record the session. Note timestamps when users hesitate, express confusion, or take unexpected paths.
  • Debrief after, not during. Save your questions for the post-task interview. Interrupting the think-aloud disrupts the data.

Other evaluation methods

Think-aloud testing is just one tool. A complete evaluation toolkit includes:

  • Task completion analysis — Can users complete key tasks? How long does it take? What's the error rate?
  • System Usability Scale (SUS) — A standardized 10-question survey that produces a score from 0–100. A SUS score above 68 is considered above average. Administer it after each test session.
  • A/B testing — Show different versions of a page to different users and measure which performs better. Even without server infrastructure, you can do manual A/B tests by alternating which version participants see.
  • Analytics review — If Bloombot had analytics (it doesn't by default), you could study page views, bounce rates, and navigation paths to identify where users get lost.

From findings to action

Raw usability data is useless without analysis. After testing, organize your findings by severity and frequency. A problem that affects every user on every task is critical; a minor annoyance noticed by one person is low priority. Create an issue log with columns: finding, severity, frequency, affected page(s), and recommended fix. This is the document that drives your next iteration.
